## v0.1 Definition

Meg v0.1 ships two user-facing capabilities, built on the same AI + prompt foundation:

| Capability | CLI | What the user gets |
|------------|-----|-------------------|
| **Generate** | `meg "…"` | A correct, ready-to-run FFmpeg command from plain English, plus a brief explanation |
| **Explain** | `meg --explain "ffmpeg …"` | A clear breakdown of an existing FFmpeg command — what each part does and why |

Both paths must feel polished, terminal-native, and reliable on real post-production and pipeline tasks. No other features ship in v0.1.

## Milestones

### Phase 0 — Bootstrap

**Goal:** A installable Python package with a working CLI skeleton.

**Deliverables:**

- Project layout per `.cursor/rules` (`meg/`, `tests/`, `pyproject.toml`)
- Typer app with `meg --help`
- Entry point: `pip install -e .` works locally
- pytest runs (even if only smoke tests initially)
- `README.md` stub pointing to vision and this roadmap

**Exit criteria:**

- [x] `meg --help` runs without import errors
- [x] `pytest` passes in CI or locally
- [x] New contributor can find `cli.py`, `prompt.py`, `providers/` in under a minute

---

### Phase 1 — Config & providers

**Goal:** Users can point Meg at their own API keys; both backends work behind one interface.

**Deliverables:**

- `config.py` — env vars + optional `~/.meg/config.toml`
- Provider abstraction in `meg/providers/base.py`
- `anthropic.py` and `openai.py` implementations
- Auto-detect provider from keys; Claude preferred if both present
- `--provider anthropic|openai` override
- Clear, actionable errors when no key is found

**Exit criteria:**

- [x] Works with only `ANTHROPIC_API_KEY` set
- [x] Works with only `OPENAI_API_KEY` set
- [x] `--provider` forces the chosen backend
- [x] `cli.py` never imports `anthropic` or `openai` directly

---

### Phase 2 — Core loop (generate)

**Goal:** Plain English in → FFmpeg command + short explanation out.

**Deliverables:**

- `prompt.py` — system prompt and message construction for **generate** mode
- Structured model output parsing (command + explanation, scannable — not chatty)
- Default CLI output: command first, then brief explanation
- `--verbose` for a more detailed explanation

**Exit criteria:**

- [x] Default output matches CLI design rules (minimal, no decoration)
- [x] Generated commands are copy-paste runnable (sensible placeholders like `input.mkv` / `output.mp4` where paths are unknown)
- [x] Explanation covers *what* and *why* without requiring FFmpeg docs

---

### Phase 3 — Explain mode

**Goal:** Paste an existing command → structured breakdown.

**Deliverables:**

- `--explain` flag wired through Typer
- Prompt path for **explain** mode (may share system context with generate, separate user template)
- Same output discipline: scannable, not conversational
- `--verbose` applies to explain output as well

**Exit criteria:**

- [x] `meg --explain "ffmpeg -i …"` produces an accurate part-by-part breakdown
- [x] Explain quality is tested with the same rigor as generate (see test prompts below)
- [x] Errors for empty/malformed input are human-readable

---

### Phase 4 — Prompt hardening

**Goal:** Reliability on real-world operations — this phase is where v0.1 succeeds or fails.

**Deliverables:**

- Curated **test prompt suite** (below) — manual review at minimum; optional pytest markers for regression later
- Iteration on `prompt.py` until the suite consistently passes review
- Guardrails in the system prompt for common FFmpeg footguns (filter syntax, stream mapping, re-encode vs copy)

**Exit criteria:**

- [x] All **generate** prompts in the suite produce commands a power user would run as-is or with trivial path edits
- [x] All **explain** prompts in the suite produce explanations a power user would trust
- [x] No systematic failures on filtergraphs, audio extraction, or delivery-style requests

> **Note:** Phases 2–4 overlap intentionally. Ship the loop early, then spend real time here before calling v0.1 done.

---

### Phase 5 — CLI polish & errors

**Goal:** The tool feels native in the terminal.

**Deliverables:**

- Consistent error messages (missing keys, API failures, timeouts, bad flags)
- Non-zero exit codes on failure
- `--verbose` behavior documented in `--help`
- No extra banners, spinners, or marketing copy in default output

**Exit criteria:**

- [x] A failed run tells the user what to do next (set key, check network, retry)
- [x] Success path never prints more than command + explanation unless `--verbose`
- [x] UTF-8 safe output on Windows (`_configure_terminal_utf8` / `_echo` in `cli.py`)
- [x] Provider requests time out after 60s with actionable message

---

### Phase 6 — Ship

**Goal:** Someone outside this repo can install and get value on the first try.

**Deliverables:**

- [x] `README.md` — install, API keys, examples for generate and explain
- [x] Package builds (`python -m build`); wheel/sdist install verified in clean venv
- [ ] PyPI package `meg-cli` live (`pip install meg-cli` from pypi.org)
- [ ] Version tagged (e.g. `v0.1.0`)
- [x] License file (MIT)

**Exit criteria:**

- [ ] Install in under 2 minutes (vision success criterion)
- [ ] First command produces a correct, usable result (vision success criterion)
- [ ] User understands output without looking up flags (vision success criterion)
- [ ] You’d recommend it to a colleague (vision success criterion)

## Test prompt suite

Use these for manual QA during Phase 4. Mark each **G** (generate) or **E** (explain). Adjust filenames/paths as needed.

### Generate (`meg "…"`)

| # | Prompt | What “good” looks like |
|---|--------|------------------------|
| G1 | Convert this MKV to H.264 MP4 with AAC audio | Sensible codecs, `-c:v libx264`, `-c:a aac`, yuv420p if relevant |
| G2 | Extract audio as 24-bit WAV | `-vn`, pcm_s24le or equivalent, no bogus video flags |
| G3 | Scale to 1920×1080, keep aspect ratio, pad with black | `scale` + `pad` or `scale` with force_original_aspect_ratio |
| G4 | Normalize loudness to -23 LUFS (broadcast) | `loudnorm` or documented two-pass approach |
| G5 | Concatenate three MP4s in order | `concat` demuxer or filter, correct `-f` / list file pattern |
| G6 | Remux only — copy all streams, change container to MP4 | Stream copy, no unnecessary re-encode |
| G7 | Burn in SRT subtitles (hardcode) | `subtitles` or `ass` filter, re-encode video if required |
| G8 | Extract frame at 00:01:30 as PNG | `-ss` placement sane, single frame, `-frames:v 1` |
| G9 | Deinterlace 1080i to 1080p | `yadif` or `bwdif`, correct field order awareness in explanation |
| G10 | Package as ProRes 422 HQ MOV | `prores_ks`, profile, pix_fmt explained |
| G11 | IMF-style deliverable (high level) | Reasonable placeholder structure; explanation acknowledges placeholders |
| G12 | H.264 CRF 18, slow preset, web streaming | `-crf`, `-preset`, possibly `-movflags +faststart` |
| G13 | Strip all audio, keep video only | `-an`, video copy or explicit re-encode as appropriate |
| G14 | Two audio tracks — map English AAC, copy video | `-map` usage correct, explanation mentions stream selection |
| G15 | Crop center 16:9 from 4:3 source | `crop` filter with sane default dimensions in explanation |

### Explain (`meg --explain "…"`)

| # | Command (abbreviated) | What “good” looks like |
|---|----------------------|------------------------|
| E1 | `ffmpeg -i in.mp4 -vf scale=1920:1080 -c:v libx264 out.mp4` | scale, codec, single output |
| E2 | `ffmpeg -i in.mov -c copy out.mp4` | remux vs transcode called out |
| E3 | `ffmpeg -i in.mp4 -vn -acodec pcm_s24le out.wav` | audio extract, `-vn` |
| E4 | `ffmpeg -f concat -safe 0 -i list.txt -c copy out.mp4` | concat demuxer, list file |
| E5 | `ffmpeg -i in.mp4 -filter_complex "[0:v]yadif[outv]" -map "[outv]" …` | filtergraph, map labels |
| E6 | `ffmpeg -i in.mp4 -map 0:v:0 -map 0:a:1 -c copy out.mp4` | stream mapping |
| E7 | `ffmpeg -i in.mp4 -af loudnorm=I=-23:TP=-1:LRA=7 out.mp4` | loudnorm parameters |
| E8 | `ffmpeg -hwaccel cuda -i in.mp4 -c:v h264_nvenc out.mp4` | hwaccel vs encoder (explanation only — no need to recommend NVENC in generate) |

## Explicitly out of scope (v0.1)

Do not implement or scaffold these for the first release:

- GUI or web UI
- Batch processing / folder scripts
- Workflow automation or preset library
- **Error interpreter** (parse FFmpeg stderr after a failed run)
- Workflow memory / per-project preferences
- Hardware acceleration detection or recommendations
- Filtergraph visualizer, validation layer, benchmark mode

Post–v0.1 ideas live in [VISION.md](../VISION.md) under “Roadmap Candidates.”
